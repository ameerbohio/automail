# Deploy Checklist — first Proxmox bring-up

The ordered steps to take a fresh clone to a working Automail stack on the
Proxmox VM. Written to be followed top-to-bottom exactly once, on a new host.

**Scope of the three ops docs, so you know which one you want:**

| Doc | Answers |
|---|---|
| [proxmox-vm-setup.md](proxmox-vm-setup.md) | How do I create and provision the VM itself? (Proxmox wizard, Ubuntu, Docker Engine) |
| **this doc** | The VM exists — what do I do, in what order, to get a working stack? |
| [cups-host-setup.md](cups-host-setup.md) | How do I turn on real paper output? (owner-gated, last) |

**Verification is automated.** `make deploy-smoke` runs the production-profile
stack and asserts what a checklist item cannot — the edge certificate and SNI,
the security headers, the browser-reachable upload URL, the guest rate limit, and
a real encrypted job going all the way to `delivered` with `/dev/shm` clean
afterwards. It shares its preflight with §0–§2 below, so the checklist cannot rot
into aspiration ([scripts/deploy/smoke.sh](../scripts/deploy/smoke.sh)).

> ⚠️ **`make deploy-smoke` is destructive.** It runs `docker compose down -v`
> before and after, destroying the Postgres/MinIO/Redis volumes, and it seeds
> fixture rows including an admin account whose password is published in this
> repo. Run it **before** §5 (provisioning), or on a throwaway host — never
> against a stack holding real data. It refuses to start if it finds an existing
> Postgres volume unless you pass `ALLOW_DESTRUCTIVE=1`.

## 0. Host prerequisites

- [ ] **Docker Engine ≥ 27.0** + Compose v2 plugin installed
      ([proxmox-vm-setup.md §4](proxmox-vm-setup.md)). The version floor is not
      cosmetic: the Traefik service sets `DOCKER_API_VERSION=1.46`, which ships
      with Engine 27.0 and **disables** version negotiation. On Engine 24–26
      (API 1.43–1.45) the Docker provider is rejected as "too new" and every
      request 404s. Check with `docker version --format '{{.Server.Version}}'`.
- [ ] **Traefik image vs. Engine version.** `docker-compose.yml` pins
      `traefik:v3.6`. `traefik:v3.0` vendors a Docker client too old to negotiate
      an API version with Engine 29.x; the provider handshake is rejected in a
      loop, it discovers zero containers, and because **all routers are Docker
      labels**, every request 404s regardless of hostname or protocol. Use
      **≥ v3.4** — the margin this project chose and tested, not a measured
      boundary (v3.1–v3.3 were never tried). Do not downgrade the pin.
- [ ] **Host CPU supports the MinIO image.** `docker-compose.yml` pins a
      `-cpuv1` MinIO tag because the default image needs the `x86-64-v2`
      micro-architecture baseline, which this project's Proxmox host (AMD A6-3600,
      2011) lacks — the container dies on boot with
      `Fatal glibc error: CPU does not support x86-64-v2`, and since
      `cloud-server` `depends_on` MinIO, that takes the whole stack down. Do not
      "modernize" the tag. Details:
      [proxmox-vm-setup.md § Host-CPU compatibility](proxmox-vm-setup.md).
- [ ] **Ports 80 and 443 are free on the host.** They are the only ports the
      stack publishes. If something already holds them, set `TRAEFIK_HTTP_PORT` /
      `TRAEFIK_HTTPS_PORT` in `.env` — but then the browser must use the
      non-standard port too.
- [ ] **Architecture:** `amd64`. No image in the stack is built or pinned for
      `arm64`, and the Go binaries are compiled `CGO_ENABLED=0` for the build
      platform. Check with `uname -m` → `x86_64`.

## 1. Clone and generate secrets

Nothing sensitive is in git (`.env`, `*.pem`, `certs/` are gitignored) —
everything is generated on the host.

```sh
git clone <automail-repo-url> automail && cd automail

./infra/certs/gen.sh                # 1. internal mTLS PKI (cloud <-> printer)
./infra/certs/gen-jwt-keys.sh       # 2. JWT RS256 signing keypair
PRINTER_KEY_PASSPHRASE='<strong-passphrase>' \
  ./infra/certs/gen-printer-keys.sh # 3. printer document keypair (RSA-4096)
./infra/certs/gen-edge-certs.sh     # 4. browser-facing edge TLS cert
```

- [ ] All four ran. **Step 4 is not optional and is the most-missed step.**
      Traefik runs with `sniStrict: true`; if no certificate covers the routed
      hostnames it **hard-rejects every TLS handshake** and the browser shows
      `ERR_SSL_UNRECOGNIZED_NAME_ALERT` before any request reaches the portal.
      The fix is to generate the cert, **never** to disable `sniStrict`.

Two separate trust domains, deliberately not cross-wired — do not substitute one
for the other:

| | Secures | Generated by | Lives in |
|---|---|---|---|
| **Edge cert** | browser ↔ Traefik | `gen-edge-certs.sh` | `infra/traefik/edge-*.pem` |
| **Internal mTLS PKI** | cloud ↔ printer | `gen.sh` | `infra/certs/` |

The edge cert is self-signed, so browsers show a one-time "not secure" warning.
That is expected. Only a *hard* TLS rejection is a bug.

- [ ] It covers **all three** routed hostnames. Verify:
      ```sh
      openssl x509 -in infra/traefik/edge-cert.pem -noout -ext subjectAltName
      # expect: DNS:automail.local, DNS:api.automail.local, DNS:blob.automail.local
      ```
      A cert generated before `blob.automail.local` was routed will pass the
      portal check and then break uploads only.

## 2. Configure `.env`

```sh
cp .env.example .env
```

- [ ] Every `changeme*` value replaced (`openssl rand -base64 32` for the random
      ones).
- [ ] `PRINTER_KEY_PASSPHRASE` **matches** the passphrase used in step 1.3.
- [ ] `MINIO_KMS_SECRET_KEY` in `<key-name>:<base64-32-bytes>` form.
- [ ] `DEV_MODE=true` for now — physical printing is turned on last (§8).

| Var | What |
|---|---|
| `POSTGRES_PASSWORD` | Postgres superuser password |
| `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD` | MinIO console + S3 credentials |
| `MINIO_KMS_SECRET_KEY` | SSE-S3 key, `automail-sse-key:$(openssl rand -base64 32)` |
| `APP_ENCRYPTION_KEY` | pgcrypto key for the encrypted PII columns |
| `PRINTER_KEY_PASSPHRASE` | decrypts the printer's document private key |
| `DEV_MAILBOX_ID` / `SLOT_ID` | this unit's identity — see §5 |
| `MINIO_PUBLIC_ENDPOINT` / `MINIO_PUBLIC_SECURE` | object storage as the *browser* sees it — see §4 |

`REDIS_PASSWORD` is listed in `.env.example` but **is not wired to anything**:
Redis runs without `requirepass` and `REDIS_URL` carries no credential. Redis is
not published to the host, so it is reachable only from the internal Docker
network — but do not read the variable as evidence that Redis is authenticated.

## 3. DNS / hosts entries

Traefik routes by **hostname**, not path. Three names, all pointing at the VM:

| Hostname | Serves |
|---|---|
| `automail.local` | the portal (web UI) |
| `api.automail.local` | the cloud API |
| `blob.automail.local` | object storage — the browser's ciphertext upload target |

- [ ] All three resolve to `<VM_IP>` **from the machine whose browser opens the
      app** — not from the VM. Either add a LAN DNS A record for each, or add one
      line to the client's hosts file (`C:\Windows\System32\drivers\etc\hosts`,
      or `/etc/hosts`):
      ```
      <VM_IP>  automail.local  api.automail.local  blob.automail.local
      ```

Missing `blob.automail.local` is silent until someone tries to send mail: the
portal loads fine and the upload fails.

## 4. Why object storage is browser-facing

Worth understanding before you debug it. By design
([plans/09-api-contracts.md](../plans/09-api-contracts.md)) the browser PUTs the
encrypted blob **straight to object storage** using a pre-signed URL — the cloud
server never receives the blob bytes. That only works if the pre-signed URL names
a host the browser can resolve, so MinIO gets its own edge hostname rather than a
published port:

- `MINIO_ENDPOINT=minio:9000` — internal, used for all server-side blob ops.
- `MINIO_PUBLIC_ENDPOINT=blob.automail.local` — used **only** to sign the
  browser's upload PUT.

- [ ] Both set consistently with the Traefik route on the `minio` service and
      with the SAN from §1. SigV4 signs the `Host` header and Traefik passes it
      through unchanged, so the signature still verifies at MinIO.
- [ ] `MINIO_CORS_ORIGIN` is the portal's origin (`https://automail.local`). The
      upload is cross-origin, so the browser preflights it.

If `MINIO_PUBLIC_ENDPOINT` is unset, `upload_url` comes back as
`http://minio:9000/...` and every guest upload fails with nothing logged
server-side — handing out the URL succeeded.

## 5. Provision the mailbox unit in Postgres

Each physical unit needs a `mailboxes` row (with the printer's **document public
key**, `infra/certs/printer-public.pem`) and a `mailbox_slots` row.

- [ ] `DEV_MAILBOX_ID` in `.env` = the `mailboxes.id` UUID.
- [ ] `SLOT_ID` in `.env` = the `mailbox_slots.id` UUID for that mailbox.

**`SLOT_ID` has no safe default.** The printer falls back to the literal string
`"slot-1"`, which never matches a real row. The cloud's dispatch eligibility
check looks the slot up by the database id, finds nothing, and refuses to
dispatch — so **every job sits in `queued` forever**, the printer looks healthy,
and nothing reports an error. If you see that symptom, check the cloud-server log
for:

```
dispatch: mailbox <id> reported no slot <id> (reported slots: [slot-1]) -- ...
```

For a throwaway/demo fixture, `scripts/e2e/seed.sh` writes a complete set of
non-production rows (building, mailbox, slot, resident, admin sender).

## 6. Bring the stack up

```sh
docker compose up -d --build     # first build compiles Go + Next.js
docker compose ps                # every service healthy
```

- [ ] All services up; MinIO shows its startup banner, not a `glibc` fatal.
- [ ] Published ports are **only** 80 and 443. Postgres, Redis, cloud-server
      (8080/8443) and printer (8444) are not published and are reachable only
      from the internal Docker network.

      Note what that does *not* mean: object storage **is** exposed, deliberately
      (§4) — MinIO's S3 API is routed at `blob.automail.local` through the same
      edge, without the rate-limit or security-header middleware (both are
      meaningless on an S3 response; the pre-signed URL is the authorization).
      Objects there are E2EE ciphertext plus SSE-S3 at rest, so this is the
      exposure `plans/02-security.md` already accepts, but the MinIO **console**
      (9001) stays unrouted and unpublished.

## 7. Verify

- [ ] **Automated:** `make deploy-smoke` — but see the warning at the top of this
      document: it destroys volumes, so run it on this host **before** §5, or not
      at all here (running it on your workstation proves the same things about the
      images and config). It covers everything below except the browser step.
- [ ] **Edge, by hand** (`--cacert`, not `-k`, so a wrong cert fails):
      ```sh
      for h in automail.local api.automail.local blob.automail.local; do
        curl -sS --cacert infra/traefik/edge-cert.pem \
          --resolve "$h:443:127.0.0.1" -o /dev/null -w "$h -> %{http_code}\n" "https://$h/"
      done
      ```
      A TLS error means the cert or its SANs (§1); a `404` means Traefik's Docker
      provider found no routers (§0).
- [ ] **In a browser:** `https://automail.local` loads, accept the self-signed
      warning, run the guest flow (upload a PDF → guest token → `/track` goes
      `submitted → dispatching → printing → delivered`). In `DEV_MODE` the printer
      logs `dev: would print` instead of emitting paper.
- [ ] **No plaintext leaked:** `docker compose exec printer ls -la /dev/shm` shows
      no `automail-*.pdf` after a delivered job.

Stuck jobs, offline mailboxes and a backing-up dispatch stream are covered in
[runbook.md](runbook.md).

## 8. Physical printing (owner-gated, last)

Real paper output is a separate step: install CUPS on the VM, add the queue, wire
the socket into the printer container, then set `DEV_MODE=false` and
`PRINTER_NAME`. Full procedure — including the CUPS-spools-to-disk security
decision that must be made first — is in
[cups-host-setup.md](cups-host-setup.md).
