# SSE Status Stream + Redis Pub/Sub Fan-Out (vs WebSocket, vs Streams)

**What it is.** `GET /jobs/:id/stream` ([services/cloud/handlers/jobs.go](../../services/cloud/handlers/jobs.go) `StreamJob`) is a Server-Sent Events endpoint: a plain HTTP response with `Content-Type: text/event-stream` that the server keeps open and writes `data: {"job_id":"...","status":"printing"}\n\n` frames into, flushing after each one. The events come from Redis: when a printer reports a job transition up its WebSocket link, the hub on whatever node holds that socket updates Postgres and `PUBLISH`es `{"status":"delivered"}` to `job:<id>:status` ([services/cloud/link/hub.go](../../services/cloud/link/hub.go) `onStatus`); every node with a browser watching that job is subscribed to the channel and relays the payload into its open HTTP response — adding `job_id` back in, since the internal payload deliberately omits what the channel name already scopes ([plans/09-api-contracts.md](../../plans/09-api-contracts.md)). On connect the handler subscribes **first**, waits for the subscribe ack, *then* reads the current status from Postgres and sends it as a snapshot event — subscribe-then-snapshot means a transition can be seen twice (harmless) but never missed, and a client connecting after `delivered` gets one event and a clean close instead of hanging.

**Why we chose it (the tradeoff/alternative).** Two separate choices here. *SSE over WebSocket:* status flow is strictly one-directional (server → browser) — the client never sends anything after the initial GET — so WebSocket's bidirectional framing, upgrade handshake, and custom keepalive/reconnect logic buy nothing. SSE is just HTTP: it passes through Traefik and any proxy untouched, auth is a normal request (JWT header or `?token=` — the guest token rides the query string because `EventSource` can't set headers), and the browser `EventSource` API reconnects automatically with backoff for free. Contrast with the printer link, which *is* a WebSocket precisely because it's bidirectional (state/status frames up, dispatch frames down) and long-lived with its own ping/pong liveness contract. *Pub/sub over Streams:* this is the inverse of the `jobs:pending` decision ([14-redis-streams-consumer-groups.md](14-redis-streams-consumer-groups.md)). A lost queue message is a lost job, so the queue needs Streams' at-least-once + PEL + XACK. A lost status event costs nothing: Postgres holds the authoritative status (written *before* the publish), and any client that missed the event — or reconnects — gets the truth from the snapshot read. Fire-and-forget fan-out to whoever is listening *right now* is exactly pub/sub's semantics, and exactly wrong for the queue. Cross-node, this is the mirror image of dispatch fan-in ([11-dispatch-fan-in-printer-link.md](11-dispatch-fan-in-printer-link.md)): both sides of the system have a connection pinned to one node (printer socket on node A, browser stream on node B), and Redis pub/sub is the bridge that makes node-pinned connections reachable from anywhere — dispatch fans **in** (any node → the one node holding the printer socket), status fans **out** (the one node that got the status frame → every node holding a watcher).

**The honest caveat.** We don't implement `Last-Event-ID`/`retry:` — SSE's built-in resume protocol, where the browser replays the last seen event ID on reconnect so the server can backfill the gap. We get away with it because job status is a **state machine, not an event log**: events are collapsible (only the latest matters), so the snapshot-on-connect fully substitutes for replay. The cost is that a reconnecting client can skip intermediate transitions entirely (never sees `printing`, lands straight on `delivered`) — fine for a tracking page, unacceptable if events were deltas that had to be applied in order (that's when you need event IDs, or Streams with `XREAD` from a cursor as the backing store instead of pub/sub). Two more production caveats: idle SSE connections get cut by proxy idle timeouts (the standard fix is a periodic `: ping` comment line, which we skip — `EventSource` reconnect + snapshot covers the prototype), and over HTTP/1.1 each SSE stream burns one of the browser's ~6 connections per origin, which is the classic argument for HTTP/2 (multiplexed streams) — Traefik terminates HTTP/2 for us, so this is the LB's problem, not the Go server's.
