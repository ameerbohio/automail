# pgcrypto for PII Encryption at Rest

**What it is.** `senders.email_enc`, `senders.name_enc`, and `residents.name_enc` ([services/cloud/db/schema.sql](../../services/cloud/db/schema.sql)) are `BYTEA` columns holding `pgp_sym_encrypt(plaintext, app_key)` output rather than plaintext `TEXT`. `pgcrypto` is a Postgres extension (`CREATE EXTENSION IF NOT EXISTS pgcrypto`) that adds symmetric/asymmetric crypto functions callable from SQL.

**Why we chose it (the tradeoff/alternative).** This is a completely separate concern from the document E2EE (plans/02-security.md §1) — the cloud server *can* decrypt this PII when it legitimately needs to (e.g. to display a sender's own name back to them), it just shouldn't leave it sitting in plaintext in the database in case of a DB dump or backup leak. The alternative, application-layer encryption (encrypt in Go before the `INSERT`, decrypt in Go after the `SELECT`), keeps `app_key` entirely inside the application process — it never appears in a SQL statement or a query log. pgcrypto's tradeoff is the opposite: the encryption logic lives conveniently in the database (one less thing for every Go code path to get right), but `app_key` necessarily travels in the SQL call itself (`SELECT pgp_sym_encrypt($1, $2)`), meaning it's visible to anything that can see executed queries — `pg_stat_statements`, a query logger, a misconfigured monitoring tool.

**The honest caveat.** pgcrypto protects PII *at rest* against someone who gets a raw database dump or steals disk/backup media without `app_key` — it does nothing against an attacker who has live access to the database connection itself, since at that point they can just run the same `pgp_sym_decrypt` call the application does. It's also a different guarantee from the document E2EE entirely: even a fully compromised cloud server (with `app_key` in hand) can decrypt sender names and emails, but still cannot decrypt a single document, because the document's AES key was never sent to the cloud server in any form it can read.
