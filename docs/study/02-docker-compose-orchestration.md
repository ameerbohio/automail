# Docker Compose Multi-Service Orchestration

**What it is.** [docker-compose.yml](../../docker-compose.yml) declares every service in the stack (Traefik, cloud-server, portal, Postgres, Redis, MinIO) as one unit, on a shared Docker bridge network (`automail`). Containers reach each other by service name (`postgres`, `redis`, `minio`) — Docker's embedded DNS resolves the name to whatever IP that container currently has, so the address never has to be hardcoded.

**Why we chose it (the tradeoff/alternative).** For a single-VM prototype, Compose gives the same service-discovery and network-isolation properties as Kubernetes at a fraction of the operational overhead — one YAML file, one `docker compose up`. Kubernetes would add real value at multi-node scale (the production target of ~12M mailbox units obviously isn't one VM), but for a Proxmox-VM-hosted demo it would be pure ceremony: a control plane, manifests, and a learning curve that buys nothing here.

**How services stay internal.** Only Traefik publishes ports to the host (`80`, `443`); every other service has no `ports:` mapping, so it's reachable *only* from other containers on the `automail` network — not from the LAN or the internet. This is what "internal only" means in the [01-architecture.md](../../plans/01-architecture.md) topology diagram: the trust boundary is enforced by Docker's network model, not just convention.

**The honest caveat.** Compose's default bridge network has no policy enforcement beyond "on this network or not" — any container on `automail` can talk to any other container on `automail` (Postgres, Redis, MinIO all included), since there's no further network segmentation. A compromised cloud-server container could, in principle, reach Postgres directly over the network even without credentials leaking. Real defense-in-depth here is the Postgres/Redis/MinIO auth itself, not network topology — Compose's bridge network blocks the *outside world*, not lateral movement *inside* the stack.
