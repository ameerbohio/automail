# Dispatch Fan-In: Printer Dials Out, Cloud Routes Across Nodes

**What it is.** The printer is a WebSocket *client*, not a server: on boot it dials `wss://cloud-server:8443/internal/printer-link` ([services/printer/wsclient.go](../../services/printer/wsclient.go)) and holds that one connection open for its whole lifetime. The cloud server accepts the upgrade in [services/cloud/handlers/printerlink.go](../../services/cloud/handlers/printerlink.go), which hands off to [`link.Hub.Accept`](../../services/cloud/link/hub.go): it reads the printer's `register` frame, stores the live `*websocket.Conn` in a per-node [`Registry`](../../services/cloud/link/registry.go) (`mailbox_id -> conn`, in-process only), and subscribes to the Redis channel `mailbox:<id>:dispatch`. Any cloud-server node — not just the one holding the socket — can now route a job to this printer by publishing to that channel; the owning node's `pumpDispatch` goroutine reads the Redis message and writes it down the wire as a `dispatch` frame.

**Why we chose it (the tradeoff/alternative).** A printer behind NAT can't accept inbound connections, so the dispatch model has to invert: the printer dials out, the cloud holds the socket open and pushes jobs down it. That's a single-node problem on its own, but the cloud server runs as **N stateless nodes** behind a load balancer — and a TCP/WebSocket connection is fundamentally pinned to whichever one node accepted it. `tryDispatch()` (Phase 4) might run on a different node than the one holding the printer's socket. Redis pub/sub is the bridge: `PUBLISH mailbox:<id>:dispatch` reaches whichever node is subscribed (the owner), regardless of which node initiated the publish. This is the *exact mirror image* of the SSE status relay (`job:<id>:status`, Phase 5), which solves the same problem for the other direction — routing a status update to whichever node holds the *sender's* browser connection. Same primitive, opposite direction: dispatch is fan-**in** (many nodes -> one printer socket), SSE is fan-**out** (one event -> the one node holding a given browser's stream, though in principle many browsers could watch the same job).

**The honest caveat.** `Registry` is explicitly *soft state*: if the owning node crashes, its entry simply vanishes — there's no handoff. The printer's reconnect-with-backoff (whichever node accepts the new socket becomes the new owner) is what restores routability, not anything in the registry itself. The real source of truth for "is mailbox X reachable" is the Redis `mailbox:<id>:state` key's TTL (90s, refreshed by every register/state frame), not whether some node's in-memory map has an entry — a node's map entry and the TTL'd Redis key can briefly disagree (e.g. immediately after a crash, before the TTL expires), and `tryDispatch`'s `PUBLISH ... ; if receivers == 0` check is the actual proof a socket is live *right now*, deliberately not trusting the cache alone.
