# Audit Log Immutability via Postgres Trigger

**What it is.** [services/cloud/db/schema.sql](../../services/cloud/db/schema.sql) defines `audit_no_mutate()`, a trigger function that unconditionally raises an exception, and attaches it `BEFORE UPDATE OR DELETE ON audit_events`. Any attempt to modify or remove a row in that table — from the application, from a direct `psql` session, from anything connecting as a non-superuser — fails at the database engine level before the mutation happens.

**Why we chose it (the tradeoff/alternative).** The alternative is an application-level rule ("our Go code just never calls UPDATE/DELETE on this table"). That's weaker because it only holds as long as the application code is trusted — a bug, a compromised cloud-server process, or someone with direct database access can bypass it trivially. Enforcing the rule *inside* Postgres, below the application layer, means the guarantee holds even if the application is fully compromised: the attacker would need database-level privileges to tamper with history, not just application-level access.

**The honest caveat.** A Postgres superuser (or the table owner) can still `DROP TRIGGER audit_immutable` and then freely mutate the table — this trigger stops accidental or application-level tampering, not a fully privileged insider. True tamper-*evidence* (not just tamper-*resistance*) needs two more things this prototype doesn't yet have: privilege separation (`REVOKE UPDATE, DELETE ON audit_events FROM` the app's database role, so the app literally cannot drop the trigger even if compromised) and hash-chaining each row to the previous one's hash, so that even a successful deletion would be detectable by a broken chain.
