# SELECT FOR UPDATE NOWAIT: The Double-Dispatch Guard

**What it is.** Before a job goes out, `claimJob` ([services/cloud/dispatch/route.go](../../services/cloud/dispatch/route.go)) opens a Postgres transaction and runs two statements in sequence: `LockJobForDispatch` (`SELECT ... FROM jobs WHERE id = $1 AND status IN ('submitted','queued') FOR UPDATE NOWAIT`) followed by `MarkJobDispatching` (`UPDATE jobs SET status = 'dispatching' WHERE id = $1`), then commits. `FOR UPDATE` takes a row lock for the duration of the transaction; `NOWAIT` is what makes it interesting — if another session already holds that lock, Postgres returns immediately with error `55P03` (`lock_not_available`) instead of blocking the caller until the lock frees up. `claimJob` treats that error (and a plain "no rows" — already claimed or in a terminal state) the same way: `claimed = false`, not an error. `attemptDispatch` reads that as "not this node's job to dispatch," and moves on.

**Why we chose it (the tradeoff/alternative).** Two cloud-server nodes can legitimately race for the same job — a `mailbox:<id>:available` event fires on every node's dispatcher goroutine at once, and the Redis Stream's consumer group already guarantees only one of them gets a *given queued entry* from `XREADGROUP`, but the **immediate-dispatch** path (`TryDispatch`, called straight from `CreateJob`) and a **retry** path (`Retry`, from the dispatcher sweep) can still end up looking at the same job at close to the same moment — e.g. right after `RequeueJob` reverts a failed publish, or during the brief window where Redis's cached printer state and the Stream's view of the world disagree. The Postgres row lock is the system's actual source of truth for "who is allowed to move this job to `dispatching`," independent of whatever Redis currently believes. `NOWAIT` specifically (over the alternative, `SKIP LOCKED`) was a deliberate choice: `SKIP LOCKED` would silently skip the contended row and move on to the *next* one in a multi-row scan — fine for a worker pool grabbing arbitrary work, wrong here, where there's exactly one row in question and "someone else has it" is itself useful information to act on (return immediately rather than queue a connection waiting on a lock that might be held by a slow transaction). `NOWAIT` fails fast and the caller decides what "fast" means for it.

**The honest caveat.** This is **belt-and-suspenders** alongside the Redis Stream, not redundant with it — they guard different failure windows. The consumer group decides *which node* picks up a *queued* entry; the row lock decides whether *this specific commit* is allowed to happen at all, including for the immediate-dispatch path that never touches the Stream. If an interviewer asks "why do you need both," the honest answer is that Redis state (the printer cache, the Stream's delivery bookkeeping) and Postgres state (the job's actual `status` column) are two systems that can be briefly inconsistent with each other — a crash between "Redis says delivered" and "Postgres committed" is exactly the kind of gap a single source of truth (the row lock) closes, while a Redis-only design would have no such backstop. The corresponding revert path (`RequeueJob`, called by `revert()` when a publish finds zero subscribers or a downstream step fails) matters too: a claim that isn't followed all the way through to a successful publish must be undone, or the job is stuck in `dispatching` forever with nothing driving it forward — `attemptDispatch` calls `revert` on every failure branch after a successful claim, precisely so a half-finished attempt doesn't leave the row in a status nothing will ever transition out of.
