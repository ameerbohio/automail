# Redis Streams + Consumer Groups vs Pub/Sub (Job Queue)

**What it is.** `jobs:pending` is a Redis Stream — an append-only log of entries, each with an auto-assigned ID (`<ms>-<seq>`) — that backs the queue for jobs that can't dispatch immediately ([services/cloud/dispatch/route.go](../../services/cloud/dispatch/route.go) `enqueue`, called from `TryDispatch` when `attemptDispatch` returns `blocked`). Every cloud-server node creates the same consumer group, `dispatchers`, on startup (`Dispatcher.EnsureGroup`, [services/cloud/dispatch/dispatcher.go](../../services/cloud/dispatch/dispatcher.go)) — `XGROUP CREATE jobs:pending dispatchers $ MKSTREAM`, with `BUSYGROUP` (group already exists) swallowed since every node races to create it. A consumer group gives **exactly one consumer** a given entry: `XREADGROUP GROUP dispatchers <node-id> STREAMS jobs:pending >` hands out only entries no consumer in the group has seen yet, and the entry sits in that group's **Pending Entries List (PEL)** until the consumer calls `XACK`. `Dispatcher.Run` reacts to two triggers — a `mailbox:<id>:available` pub/sub event (printer just went idle) or a periodic ticker — and both call `drain`, which `XREADGROUP`s a bounded number of batches and re-attempts dispatch for each via `Retry` (not `TryDispatch` — see the caveat below).

**Why we chose it (the tradeoff/alternative).** Plain Pub/Sub was the obvious first idea — the dispatch fan-in itself (`mailbox:<id>:dispatch`) and the SSE fan-out (`job:<id>:status`) both use it, and it's tempting to reuse the same primitive for the job queue. But Pub/Sub has no memory: a message published while no one is subscribed is gone, and there's no acknowledgement — if the one subscriber that received it crashes before finishing, the message is simply lost, not retried. That's fine for "tell whichever live node holds this socket" (the socket itself is the durable thing; if it's gone the printer reconnects and resubscribes), but it's wrong for "exactly one node should claim this queued job and not lose it if that node dies mid-claim." A Stream's consumer group gives **at-least-once delivery**: the PEL is the queue's memory of "this entry was handed out but not yet confirmed done," and nothing disappears just because the consumer who picked it up disappeared too. The group also solves the *which node* problem cleanly — `N` dispatcher goroutines (one per cloud-server node) all read from the same group, and Redis guarantees no two of them get the same entry from a fresh `XREADGROUP >` read, without any of them coordinating directly.

**The honest caveat.** "At-least-once delivery" is not "exactly-once processing" — those are different guarantees, and conflating them is the kind of thing an interviewer will probe. A message can be delivered, partially processed, and then redelivered (via `XAUTOCLAIM` after a crash, see [15-select-for-update-nowait.md](15-select-for-update-nowait.md) for the lock that makes redelivery safe) — the Stream guarantees the entry isn't *lost*, not that `attemptDispatch` only ever runs once for it. That's why `Retry` (the dispatcher's re-attempt path) deliberately differs from `TryDispatch` (the immediate, first-attempt path): `TryDispatch` calls `enqueue` (a fresh `XADD`) when blocked, because nothing is in the stream for that job yet; `Retry` must **not** re-`XADD` on a still-blocked outcome, because the entry being retried is already in the PEL — XADD-ing a duplicate on every `available` event for a chronically-offline printer would make the backlog grow without bound. Instead a still-blocked `Retry` simply leaves the message un-ACK'd; the next `available` event or the periodic `XAUTOCLAIM` sweep (`MinIdle` 60s, matching `plans/03-scaling.md`'s example) picks it back up. The actual "is this the same job being double-finalized" guard isn't the Stream at all — it's the Postgres row lock, the next note.
