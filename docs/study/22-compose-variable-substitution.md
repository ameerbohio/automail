# Docker Compose Variable Substitution: `${VAR-x}` vs `${VAR:-x}`

**What it is.** Compose YAML supports shell-style interpolation inside string values: `${VAR}`, `${VAR:-default}`, `${VAR-default}`, and the `?`-error variants. These are resolved once, when `docker compose` parses the file, against the invoking shell's environment (plus `.env`). A value with **no** `${...}` in it — a plain literal like `CUPS_SERVER: cups` — is not templated at all; nothing in the shell environment can change what ends up in the container, no matter what gets `export`ed before `docker compose up` runs.

**The two defaulting forms are not interchangeable.** Both fall back to `default` — but under different conditions:
- `${VAR:-default}` — POSIX "use default if VAR is unset **or empty**."
- `${VAR-default}` — POSIX "use default only if VAR is **unset**"; an explicitly-set empty string (`export VAR=""`) is preserved as `""`.

This repo's `docker-compose.demo-print.yml` shipped `CUPS_SERVER: cups` as a bare literal. `scripts/demo/up.sh`'s `PRINT=host` mode tries to redirect `lp` back to the real host printer by doing `export CUPS_SERVER=""` before bringing the stack up — the intent being "empty means fall back to the mounted `/run/cups/cups.sock`" (see [12-mtls-websocket-client-server.md](12-mtls-websocket-client-server.md) for the socket-mount half of this path, and [../cups-host-setup.md](../cups-host-setup.md) for the full real-printing setup). Because the compose file used a literal instead of `${CUPS_SERVER:-cups}` or `${CUPS_SERVER-cups}`, that `export` was inert: the container always got `CUPS_SERVER=cups`, silently routing every `PRINT=host` job at the ephemeral demo `cups` container (default backend `tofile:/out/last-print.pdf`) instead of the real printer. Three jobs reported `delivered` — genuinely true, `lp` really did succeed — just against the wrong CUPS server. Fixed to `CUPS_SERVER: ${CUPS_SERVER-cups}`: unset (the `PRINT=1` path, which never touches this variable) still defaults to `cups`; explicitly-empty (the `PRINT=host` path) now survives instead of being treated as "unset."

**Why this class of bug is easy to miss.** Nothing errored. `lp` exited 0 (it *did* reach a CUPS server — just not the intended one), the app's own success-path logging is intentionally silent (see `services/printer/print.go` — only the failure and dev-mode branches log), and the fake destination happened to be named identically to the real queue (`Canon_MF240`), so even `lpstat -p Canon_MF240` run inside the printer container looked correct. The only way to catch it was cross-referencing the *host's* CUPS job history against the container's — the two `Canon_MF240` queues live in different processes entirely.

**The honest caveat.** `${VAR-default}` only helps when the override is meant to be a real, distinguishable value like empty-string. If the desired override *is* itself empty and indistinguishable from "not set" for the consumer (e.g., a flag that's either absent or `"1"`), this pattern doesn't disambiguate anything — you'd need a separate presence-check variable. It solved this specific case only because `CUPS_SERVER=""` (fall back to socket) is semantically different from `CUPS_SERVER` unset-and-defaulted (talk to the demo container), and `lp` itself treats an empty `CUPS_SERVER` as "use the default socket," not as an error.
